# OpenShare 状态流转设计

本文档对应阶段二 `2.3`，只定义当前主链路所必需的三类状态流转：

- `submission`
- `resource`
- `report`

所有状态变化统一由 `service` 层控制，`handler` 不直接改状态，模型层不依赖 hook 承载业务流转。

## 设计原则

- `submission` 审核状态与 `resource` 公开状态严格分离
- 状态变更要同时考虑数据库与文件系统一致性
- 涉及管理员决策的状态变化，默认需要记录管理员身份和操作日志
- 需要解释“为什么”的流转，必须有原因字段

## 1. `submission` 状态流转

状态集合：

- `pending`
- `approved`
- `rejected`

### 1.1 初始创建：`pending`

触发人：

- Guest 用户
- 具备上传权限的用户
- 后续如支持本地导入，也可由系统任务触发

触发时机：

- 文件上传到 `staging` 成功
- `submission` 记录创建成功

修改字段：

- `status = pending`
- `receipt_code`
- `title_snapshot`
- `description_snapshot`
- `uploader_ip`
- `created_at`
- `updated_at`
- `reviewer_id = null`
- `reviewed_at = null`
- `reject_reason = null`

是否需要记录原因：

- 不需要

是否需要写操作日志：

- 不需要写管理员操作日志
- 是否记录访客上传审计，留到上传链路实现时单独决定

### 1.2 审核通过：`pending -> approved`

触发人：

- Admin
- Super Admin

触发时机：

- 管理员审核通过上传资料
- 数据库更新和文件从 `staging` 移入 `repository` 必须纳入同一业务事务编排

修改字段：

- `status = approved`
- `reviewer_id`
- `reviewed_at`
- `updated_at`
- `reject_reason = null`

关联动作：

- 创建或更新对应 `file` 记录
- 将对应资源状态设置为 `active`
- 执行文件移动或重命名

是否需要记录原因：

- 不强制

是否需要写操作日志：

- 需要

### 1.3 审核驳回：`pending -> rejected`

触发人：

- Admin
- Super Admin

触发时机：

- 管理员审核拒绝上传资料

修改字段：

- `status = rejected`
- `reviewer_id`
- `reviewed_at`
- `updated_at`
- `reject_reason`

关联动作：

- 暂存文件保留、删除还是移入 `trash`，在上传审核实现阶段再定具体策略
- 不创建公开 `file` 记录；若已产生中间记录，必须回滚或标记不可用

是否需要记录原因：

- 需要，`reject_reason` 必填

是否需要写操作日志：

- 需要

### 1.4 禁止的流转

- `approved -> pending`
- `rejected -> pending`
- `approved -> rejected`
- `rejected -> approved`

说明：

- 当前方案下，`submission` 审核结论一旦产生即不可反复横跳
- 若后续需要“重新审核”，应新建显式流程，而不是直接覆写历史结论

## 2. `resource` 状态流转

适用对象：

- `files`
- `folders`

状态集合：

- `active`
- `offline`
- `deleted`

### 2.1 创建公开资源：`active`

触发人：

- Admin
- Super Admin
- 后续本地导入任务

触发时机：

- 上传审核通过
- 或本地导入成功并允许公开

修改字段：

- `status = active`
- `updated_at`
- 若是首次创建，则写入 `created_at`
- `deleted_at = null`

关联动作：

- 资源可被浏览、搜索、下载

是否需要记录原因：

- 不强制

是否需要写操作日志：

- 需要

### 2.2 下架资源：`active -> offline`

触发人：

- Admin
- Super Admin

触发时机：

- 举报审核成立但选择暂不删除
- 管理员主动下架资源
- 发现文件损坏、内容不当或需要临时隐藏

修改字段：

- `status = offline`
- `updated_at`

关联动作：

- 资源不再公开展示
- 资源不可下载
- 文件通常仍保留在 `repository`，以便后续恢复或复核

是否需要记录原因：

- 需要，建议统一保存在后续的 `offline_reason` 或操作日志详情中

是否需要写操作日志：

- 需要

### 2.3 恢复资源：`offline -> active`

触发人：

- Admin
- Super Admin

触发时机：

- 举报驳回后恢复展示
- 管理员确认资源可重新公开

修改字段：

- `status = active`
- `updated_at`

关联动作：

- 资源重新进入公开列表、搜索和下载路径

是否需要记录原因：

- 不强制，但建议记录恢复说明到操作日志

是否需要写操作日志：

- 需要

### 2.4 删除资源：`active -> deleted` 或 `offline -> deleted`

触发人：

- Super Admin
- 被授权具有删除权限的 Admin

触发时机：

- 管理员确认永久下线资源
- 举报成立且决定删除
- 清理无效内容或重复资源

修改字段：

- `status = deleted`
- `updated_at`
- `deleted_at`

关联动作：

- 优先将磁盘文件或目录移入 `trash`
- 默认不再公开展示、不参与搜索、不允许下载

是否需要记录原因：

- 需要

是否需要写操作日志：

- 需要

### 2.5 禁止的流转

- `deleted -> active`
- `deleted -> offline`

说明：

- 当前阶段把 `deleted` 视为业务终态
- 如果未来要支持从 `trash` 恢复，应单独设计“恢复删除资源”的治理流程，并审查其对目录唯一键和文件路径的影响

## 3. `report` 状态流转

状态集合：

- `pending`
- `approved`
- `rejected`

### 3.1 创建举报：`pending`

触发人：

- Guest 用户

触发时机：

- 用户提交举报表单成功

修改字段：

- `status = pending`
- `file_id` 或 `folder_id`
- `created_at`
- `updated_at`
- `reviewer_id = null`
- `reviewed_at = null`

关联动作：

- 被举报资源状态不立刻变化
- 进入后台待审核队列

是否需要记录原因：

- 需要，举报原因在记录创建时必须写入

是否需要写操作日志：

- 不需要写管理员操作日志

### 3.2 举报成立：`pending -> approved`

触发人：

- Admin
- Super Admin

触发时机：

- 管理员审核认定举报有效

修改字段：

- `status = approved`
- `reviewer_id`
- `reviewed_at`
- `updated_at`
- `review_reason`

关联动作：

- 目标资源通常切换为 `offline`
- 如决定直接删除，则目标资源进入 `deleted`

是否需要记录原因：

- 需要，审核说明必填

是否需要写操作日志：

- 需要

### 3.3 举报驳回：`pending -> rejected`

触发人：

- Admin
- Super Admin

触发时机：

- 管理员审核认定举报不成立

修改字段：

- `status = rejected`
- `reviewer_id`
- `reviewed_at`
- `updated_at`
- `review_reason`

关联动作：

- 若资源此前因预防策略已被下架，需显式决定是否恢复为 `active`
- 默认情况下，单纯举报驳回不改资源状态

是否需要记录原因：

- 需要，审核说明必填

是否需要写操作日志：

- 需要

### 3.4 禁止的流转

- `approved -> pending`
- `rejected -> pending`
- `approved -> rejected`
- `rejected -> approved`

说明：

- 举报审核结论也视为一次性结论
- 如果后续需要复议机制，应该新增复议记录，而不是直接重写原举报状态

## 4. 操作日志要求总览

必须写入 `operation_logs` 的动作：

- 审核通过 `submission`
- 驳回 `submission`
- 下架 `resource`
- 恢复 `resource`
- 删除 `resource`
- 审核通过 `report`
- 驳回 `report`

默认不写管理员操作日志的动作：

- 创建 `submission`
- 创建 `report`

说明：

- 非管理员触发动作是否额外写独立审计，放到对应功能实现阶段单独设计
- 操作日志至少需要能记录操作人、动作类型、目标对象、结果、原因摘要

## 5. 一致性要求

### `submission -> resource`

- `submission` 审核通过前，不允许资源公开
- `submission` 审核通过后，必须保证数据库状态更新与文件移动逻辑整体成功，不能出现“数据库已 approved，但文件仍停留在 staging”的最终状态
- 如果文件移动失败，`submission` 不得落成最终 `approved`

### `report -> resource`

- `report approved` 不强制等于 `resource deleted`
- 举报成立后的最小动作是把目标资源切到 `offline`
- 是否进一步删除，由管理员权限和治理策略决定

### `deleted` 终态

- `resource.status = deleted` 后，默认视为不可直接恢复
- 若将来支持恢复，需要新增明确流程和补偿逻辑
